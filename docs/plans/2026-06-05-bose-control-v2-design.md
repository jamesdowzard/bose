# Bose Control v2 — Design

**Date:** 2026-06-05
**Status:** Approved (design phase)
**Scope:** Full from-scratch rebuild with a single-source-of-truth protocol + codegen.

## Why

The current project controls Bose QC Ultra 2 headphones over Bluetooth RFCOMM (BMAP)
from a macOS app, a CLI, and an Android app. It works, but it has accreted into a mess.
This rebuild was triggered by a real-world symptom: **audio dropping out for ~30s every
10–20 minutes**, plus the phone's Bose Music app being unable to connect.

### Root-cause audit findings

1. **Dropout cause — a background poll that violates the project's own rules.**
   `macos/BoseControl/BoseManager.swift:50` polls the headphones **every 10s, 24/7**
   (`pollInterval = 10.0`), started at `BoseApp.swift:13` on window-appear (never stopped)
   and kept resident by the `KeepAlive` LaunchAgent. Each tick runs a heavy multi-command
   `getAllState()` RFCOMM session. The repo's own `CLAUDE.md` states: *"RFCOMM opens ACL…
   Don't probe/poll from a device that isn't supposed to be the active source"* and
   *"No persistent connections."* Commit #62 (`7a78269`) removed the 30s auto-**reconnect**
   timer for exactly this reason, but the 10s state **poll** was left behind — a
   half-finished fix. On a 2-slot multipoint headphone, the Mac holding a slot and hammering
   RFCOMM every 10s contends with the phone's active A2DP stream; the firmware periodically
   re-routes the link → the ~30s dropout. RFCOMM/SPP is single-session, so the Mac cycling it
   also blocks the phone's Bose Music app.

2. **The structural mess — the BMAP wire protocol is hand-maintained twice.**
   Swift `BoseRFCOMM.swift` (972 LOC) and Kotlin `BoseProtocol.kt` (628 LOC) are two
   independent copies of the same reverse-engineered byte protocol. They have already
   drifted: Swift `setVolume` sends the wrong operator (`OP_SET 0x06` instead of
   `SET_GET 0x02`) and is silently broken — the CLI only works because it bypasses the
   method with raw bytes. The `tv` device exists in some maps but not others.

3. **macOS app is the weakest surface.** A *window* app (`WindowGroup`, Dock icon) running
   headless under a LaunchAgent. ~10 fully-implemented `BoseManager` methods have **no UI**
   (CNC depth, multipoint, media transport, disconnect, rename). A dozen state fields are
   fetched every poll and never displayed. No `stopPolling()` is ever called.

4. **Protocol correctness bugs.** `connectDevice` treats ACK as success (violates the
   documented ACK≠success rule — should poll `getConnectedDevices`). A single shared mutable
   `RFCOMMDelegate` is unsafe for an always-on app. Multi-packet BMAP responses can be parsed
   mid-stream (no length framing). `getActiveDevice` (documented unreliable) is still consumed.

5. **Config drift.** The headphone MAC is hardcoded as separate string literals in 5+ places
   across both platforms (colon vs dash forms in Swift alone). Device maps differ between
   surfaces.

6. **Dead weight.** `probe-gatt.swift` is an abandoned BLE-GATT wrong-turn that the repo's own
   docs disproved. Stale/lying comments in both apps (e.g. BoseManager header claims
   "auto-reconnect"; Android logs "confirmed by RESULT frame" when it polls).

### What is worth keeping (do NOT throw away)

- **The reverse-engineered BMAP knowledge** in `CLAUDE.md` — months of captures (operators,
  ACK≠success, HFP-blocks-A2DP, companion-device, the #61–64 auto-reconnect saga). This is the
  irreplaceable asset and becomes the basis of the spec.
- **The Android architecture** — foreground service, companion-device registration, home
  widget, QS tile, A2DP media-nudge. It works; only its protocol layer gets regenerated.
- **The Swift RFCOMM lifecycle** — `withRFCOMM` open/drain/close, cold-start warm-up
  (error 913), single-session bulk queries. The transport pattern is sound.

## Approach (approved)

**Full from-scratch + codegen**, with an **80% codegen split** and a **Python + TOML**
toolchain. Define the BMAP protocol once; generate the per-platform protocol layers; rebuild
the Mac app; regenerate Android's protocol on its existing architecture; regenerate the CLI.

## Architecture

```
bose/
  protocol/
    spec/bmap.toml          # SINGLE SOURCE OF TRUTH: every command, operator, enum
    spec/devices.toml       # headphone MAC + device map (replaces 5+ hardcoded literals)
    codegen/generate.py     # uv-run Python; TOML -> Swift + Kotlin
    generated/BMAP.generated.swift
    generated/BMAP.generated.kt
    tests/test_golden.py    # asserts generated bytes == verified captures (pytest)
  macos/                    # rebuilt event-driven menu-bar app (native transport + generated)
  android/                  # rebuilt on generated Kotlin (keep FGS/widget/companion arch)
  cli/                      # bose-ctl on generated Swift
  docs/
```

### The codegen boundary (the 80% split)

**Codegen owns** the declarative wire details — exactly where drift happens:
- Frame construction (block / function / operator / length).
- Scalar get/set commands and their payload encoders.
- Enums (ANC modes, EQ bands, media actions).
- The device map and headphone config (from `devices.toml`).

Example spec entry:

```toml
[commands.anc_mode]
block = 0x1F
function = 0x03
get = { operator = "GET" }
set = { operator = "START", payload = ["mode:u8", "0x01"] }
enum = "AncMode"
verified_bytes = { set_quiet = "1F 03 05 02 00 01" }   # golden test locks this

[enums.AncMode]
quiet = 0
aware = 1
custom1 = 2
custom2 = 3
```

→ emits matching Swift + Kotlin frame builders / decoders + a golden test asserting the
emitted bytes equal the verified capture.

**Hand-written per platform (calling generated primitives):**
- **Transport** — IOBluetooth (Swift) vs `BluetoothSocket` (Kotlin) are irreducibly native:
  open RFCOMM, 300ms drain, send, read, close; cold-start warm-up; serial queue.
- **~3 composite commands** that need read-modify-write or list parsing:
  - CNC / ANC depth (`1F,0A`) — read current 5-byte tuple, change level, preserve the rest.
  - Connected-device-list parsing (`05,01`) — variable-length MAC list (ground truth).
  - Bulk `getAllState` — one session issuing many generated sub-commands.

  These are marked `composite = true` in the spec so the split is explicit, not accidental.

### Correctness fixes folded into the rebuild

- `setVolume` uses `SET_GET 0x02` (spec-defined, generated correctly for both platforms).
- `connectDevice` confirms by polling `getConnectedDevices` until the target MAC appears in
  the audio-active list — never treats ACK as success.
- Transport uses a **serial queue + per-call response framing** so an always-on app is safe.
- Single device map + headphone MAC, sourced from `devices.toml`.

## Components

### macOS app — rebuilt
- **True menu-bar app**: SwiftUI `MenuBarExtra` + `.accessory` / `LSUIElement` (min target is
  already 13.0). No Dock-window mismatch.
- **Event-driven, never background-polled**: refresh on menu-open + IOBluetooth
  connect/disconnect notifications only. This structurally eliminates the dropout bug.
- **Surfaces everything**: device switch, ANC mode, CNC depth slider, volume, EQ
  presets + 3-band sliders, multipoint toggle, media transport, per-device disconnect,
  rename, and an info readout (firmware / serial / codec / auto-off).
- **Global hotkey** to cycle devices (`mac → quest → ipad → iphone → tv → phone`).
- **Honest async feedback**: per-device "connecting…" state with `getConnectedDevices`
  confirmation, failure surfaced (not optimistic-silent), revert on failure.

### Android — regenerated protocol, kept architecture
- Keep FGS, companion device, widget, QS tile, media nudge.
- Replace hand-written `BoseProtocol.kt` with generated Kotlin + a thin transport.
- Fix the mess: unify the device map (the `tv` widget drift), add media controls to the
  notification (MediaSession), isolate the hidden-API A2DP reflection hack, remove stale
  comments.

### CLI + cleanup
- `bose-ctl` regenerated on the shared Swift protocol; collapse inline byte-parsing onto
  generated methods.
- Delete `probe-gatt.swift`.

## Testing

- **Golden byte tests (pytest)**: for every spec command with `verified_bytes`, assert the
  generated builder output equals the documented capture. The existing `CLAUDE.md` corpus
  becomes the regression suite — the spec cannot silently drift from reality.
- **Round-trip decode tests** on captured response frames where available.
- **Per-platform smoke**: build the generated Swift/Kotlin; a tiny harness that constructs
  each command and checks bytes (no live headphones needed for CI-equivalent checks).
- **Manual live verification** (real headphones) before declaring each app done: switch
  devices, set ANC/volume/EQ, confirm no dropouts over a 30+ min session, confirm the phone
  app can still connect when the Mac app is idle.

## Migration / phasing

Headphones must keep working throughout. Phased so each lands independently via `/f:new`
(or sequential commits on this branch, shipped per phase):

1. **Protocol + codegen** — `bmap.toml`, `devices.toml`, `generate.py`, golden tests.
   Proven against captures before any app consumes it.
2. **macOS app** — rebuild on generated Swift + native transport. Retire the 10s poll.
   Replace the LaunchAgent semantics (menu-bar resident, not headless window).
3. **Android** — regenerate protocol; wire fixes; verify widget/tile/service still pass.
4. **CLI** — regenerate; delete `probe-gatt.swift`; final cleanup.

## Risks / notes

- **Headphone-side single-session contention is real** — only ever one active control surface
  at a time. The rebuilt Mac app must idle silently (no probing) so the phone can own the
  link, and vice versa. This is the lesson the v1 poll forgot.
- **A2DP reflection hack on Android** is OEM-fragile; isolate but don't expand it.
- **Golden tests depend on capture accuracy** — where `CLAUDE.md` bytes are unverified, mark
  them and confirm against a live capture before locking.
- **Composite commands are the codegen escape hatch** — keep the count at ~3; if the list
  grows, revisit whether the spec DSL should absorb them.
